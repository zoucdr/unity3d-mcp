# Unity3D-MCP 项目规则

## 1. 代码风格规范

### C# 代码规范
- 使用 PascalCase 命名类、方法和公共变量
- 使用 camelCase 命名私有变量和参数
- 每个类和方法应有简洁的注释说明功能
- 保持每行代码不超过 120 个字符
- 使用 4 个空格进行缩进
- 大括号使用 Allman 风格（新行开始）

### Python 脚本规范
- 遵循 PEP 8 标准
- 使用 snake_case 命名函数和变量
- 确保 Python 脚本有适当的错误处理

## 2. 资源组织

### Assets 目录结构
- 按功能模块组织子目录
- 资源文件使用明确的命名前缀（如：UI_、SFX_、Model_等）
- 确保资源命名不含特殊字符和空格

### 脚本组织
- 核心脚本放在 `Assets/Scripts` 目录
- 扩展功能放在 `Assets/Extends` 目录
- 示例代码放在 `Assets/Demos` 目录

## 3. MCP 相关规范

### MCP 配置
- MCP 配置文件存放在 `.cursor/mcp.json` 和 `.vscode/mcp.json`
- 确保 MCP 工具规则与项目需求匹配

### Python 集成
- Python 脚本统一放在 `Python/` 目录
- 材质转换相关脚本命名使用 `convert_*_mcp.py` 格式
- 确保 Python 脚本有清晰的注释说明功能和使用方法

## 4. 警告处理
- 及时清理 Unity 控制台中的警告信息
- 特别关注未使用变量（CS0219）和过时 API（CS0618）警告
- 使用 MCP 工具的 console_read 功能定期检查警告

## 5. 文件访问权限
- 遵循项目目录访问限制，避免访问工作目录外的文件
- 所有资源文件应放在项目目录内以便版本控制

## 6. IDE 配置
- VS Code 配置文件存放在 `.vscode/` 目录
- 确保 `.vscode/settings.json` 包含适当的项目设置

## 7. 提交规范
- 提交前确保代码编译通过，无警告
- 提交信息应清晰描述变更内容
- 大型功能应分多次小提交

## 8. 材质转换规则
- 使用 URP 渲染管线时，确保所有材质正确转换
- 优先使用项目中的 `convert_materials_to_urp_mcp.py` 脚本进行批量转换

## 9. 本地化规范
- 中文字符串翻译文件存放在 `Python/chinese_strings_*.csv`
- 确保新添加的 UI 文本都有适当的本地化支持

## 10. 性能优化
- 避免在 Update 方法中执行复杂计算
- 合理使用对象池管理频繁创建销毁的对象
- 优化资源引用，避免内存泄漏
